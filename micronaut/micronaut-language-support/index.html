Language Support

  • 本文由本人翻译自官方文档

Micronaut 支持任何实现了 Java Annotation Processor API 的 JVM 语言

尽管 Groovy 没有支持这个 API, 但是特别的支持已经通过使用 AST transformations 来构建

当前支持的语言有: Java, Groovy, Kotlin (通过 kapt 工具)

Theoretically any language that supports a way to analyze the AST at compile time could be supported.

理论上, 任何支持在编译时分析 AST 的语言都可以被支持

The io.micronaut.inject.writer package includes language-neutral classes that build BeanDefinition classes at compile time using the ASM tool.

io.micronaut.inject.writer 包 包括了与语言无关的类, 这些类在编译时使用 ASM 工具构建 BeanDefinition 类

接下来的部分介绍了通过特定语言使用 Micronaut 的功能和注意事项

Micronaut for Java (用于 Java 的 Micronaut)

对于 Java, Micronaut 使用一个 Java BeanDefinitionInjectProcessor 注解处理器来在编译时处理类并生成 BeanDefinition 类

这里最大的优点是在编译时付出了少量成本, 当在运行时 Micronaut 基本上是无反射的, 快速的且内存消耗非常少

Using Micronaut with Java 9+ (在 Java 9+ 中使用 Micronaut)

Micronaut 是使用 Java 8 构建的, 但在 Java 9 和更高版本中也正常运行

Micronaut 生成的类与同一个包中的现有类 并排 放置, 因此不会违反有关 Java 模块系统的任何内容

将 Java 9+ 与 Micronaut 一起使用时有一些注意事项

The javax.annotation package

如果你使用 Micronaut CLI 创建项目, 如果你使用 Java 9+, 那么 javax.annotation 依赖会自动添加到你的项目中

javax.annotation, 包含了 @PostConstruct,@PreDestroy 等, 已经从核心 JDK 迁移到一个 module 中了

一般来说, 应该避免使用这个包中的注解, 应该使用等价的 jakarta.annotation 包

Incremental Annotation Processing with Gradle (使用 Gradle 进行增量注解处理)

Micronaut 支持 Gradle incremental annotation processing, 这个处理通过只编译修改了的类, 避免完全重编译来提供构建速度

However, the support is disabled by default since Micronaut allows the definition of custom meta-annotations (to for example define custom AOP advice) that need to be configured for processing.

但是, 默认情况下禁用了该支持, 因为 Micronaut 允许定义需要被配置以进行处理的自定义元注解 (例如定义自定义 AOP advice)

Project Lombok 是一个流行的 Java 库, 通过注解处理器添加了一些有用的 AST transformations 到 Java 语言中

Micronaut for GraalVM (用于 GraalVM 的 Micronaut)

GraalVM 是一个来自 Oracle 的通用虚拟机, 支持跨语言运行时环境, 可以将 Java 应用程序编译成原生机器代码

所有 Micronaut 应用程序可以使用 GraalVM 来运行, 但是已经向 Micronaut 添加了特殊支持, 以支持使用 GraalVM’s native-image tool 运行 Micronaut 应用程序

Micronaut 现在支持 22.0.0.2 版本的 GraalVM, 开发团队正在改进每个新版本的支持

Don’t hesitate to report issues however if you find any problem.

Micronaut 的许多模块和第三方库已经通过验证可以与 GraalVM 一起使用:

  • HTTP server
  • HTTP client
  • Function support
  • Micronaut Data JDBC
  • JPA
  • Service Discovery
  • RabbitMQ
  • Views
  • Security
  • Zipkin
  • etc

Support for other modules is evolving and will improve over time.

对其他模块的支持正在发展, 将会持续改进

Understanding Micronaut and GraalVM (理解 Micronaut 和 GraalVM)

Adding Additional Classes for Reflective Access (为反射访问添加额外的类)

To inform Micronaut of additional classes to be included in the generated reflect.json file at compile time, either annotate a class with @ReflectiveAccess or @TypeHint.

为了在编译时通知 Micronaut 在生成的 reflect.json 文件中包含额外的类, 请使用 @ReflectiveAccess 或 @TypeHint

Both allows for reflective access, and the latter is typically used on a module or Application class to include classes that are needed reflectively.

这两个注解都允许反射访问, @TypeHint 通常用于模块或应用程序类以包含反射所需的类

GraalVM and Micronaut FAQ (GraalVM 和 Micronaut 常见问题解答)

How does Micronaut manage to run on GraalVM? (Micronaut 如何在 GraalVM 上运行?)

Micronaut 提供了不使用反射的依赖注入和AOP运行时. 这使得 Micronaut 应用程序更加容易运行在 GraalVM 上, 因为在原生镜像中的反射有很大的限制

How can I make a Micronaut application that uses picocli run on GraalVM? (如何让使用 Picocli 的 Micronaut 应用程序运行在 GraalVM 上)

Picocli 提供了一个带有生成 GraalVM 反射配置文件工具的 picocli-codegen 模块

这个工具可以手动运行, 也可以作为构建的一部分自动运行

这个模块的 README 包含代码片段的使用说明, 用于配置 Gradle 和 Maven 以在构建过程中自动生成 cli-reflect.json 文件

在运行 native-image 工具时添加生成的文件到 -H:ReflectionConfigurationFiles 选项中

What about other Third-Party Libraries? (那第三方类库呢)

Micronaut 不能保证第三方类库可以运行在 GraalVM SubstrateVM 上, 每个类库要自己去实现支持

What if I want to set the heap’s maximum size with -Xmx, but I get an OutOfMemoryError? (如果我想要使用 -Xmx 设置最大堆内存, 但我得到一个 OutOfMemoryError)

如果你在用来构建原生镜像的 Dockerfile 中设置了最大堆内存, 你很有可能得到一个运行时错误:

java.lang.OutOfMemoryError: Direct buffer memory

这个问题是因为 Netty 尝试使用默认配置 io.netty.allocator.pageSize 和 io.netty.allocator.maxOrder 为每个块分配 16MB 内存:

int defaultChunkSize = DEFAULT_PAGE_SIZE << DEFAULT_MAX_ORDER; // 8192 << 11 = 16MB

最简单的解决方案是在你的 Dockerfile 的 entrypoint 中指定 io.netty.allocator.maxOrder. 一个可行的例子是 -Xmx64m:

ENTRYPOINT ["/app/application", "-Xmx64m", "-Dio.netty.allocator.maxOrder=8"]

更进一步, 你可以尝试使用 io.netty.allocator.numHeapArenas 或者 io.netty.allocator.numDirectArenas. 你可以在 official documentation 中找到更多关于 Netty 的 PooledByteBufAllocator 信息
