Management & Monitoring

声明:

  • 本文由本人翻译自官方文档
  • 部分内容可能会有修改或删减, 一切以官方文档为准.
  • 本文翻译时 Micronaut 版本为 3.4.2

如果你使用 Micronaut CLI 创建项目, 使用 management 功能来配置 management endpoints 到你的项目中:

mn create-app my-app --features management

受 Spring Boot 和 Grails 的启发, Micronaut management 依赖添加了对通过 endpoints 监控你的应用程序的支持

endpoints: 一个特殊的 URIs, 返回关于应用程序的健康和状态细节

management endpoints 同时和 Micronaut 的 security 依赖集成, 允许在你的安全系统中将敏感数据限制于授权的用户 (查看 安全章节中的Built-in Endpoints Access)

要使用 management 功能, 添加如下依赖到你的构建中:

implementation("io.micronaut:micronaut-management")

Creating Endpoints (创建端点)

除了 内建的 endpoints, management 依赖还提供了自定义 endpoints 的支持

这可以像内置 endpoints 一样启用和配置, 并可用于检索和返回任何指标或其他应用程序数据

Endpoint Sensitivity (端点敏感性)

可以通过 endpoint 注解和配置来控制整个 endpoint 的敏感性

但是, 单个方法可以独立于整个 endpoint 进行配置

@Sensitive 注解可以应用到方法上来控制方法的敏感性

Example - AlertsEndpoint:

1 : endpoint 默认为不敏感的, 并且使用默认的前缀 (prefix) endpoints

2 : 这个方法总是敏感的, 不管其他任何因素

3 : property 的值被附加到前缀 (prefix) 和 id 以查找配置值

如果配置 key endpoints.alerts.add.sensitive 有进行设置, 那这个 key 对应的值将指明 addAlert 方法的敏感性

  1. endpoint 是第一个标记 (token), 因为这是 @Endpoint 注解中默认的前缀值, 并且在这个例子中没有明确指定前缀值
  2. alerts 是第二个标记 (token), 因为这是 endpoint 的 id
  3. add.sensitive 是最后一个标记 (token), 因为这是设置到 @Sensitive 注解的 property 成员的值

如果配置 key 没有设置, 会使用 defaultValue (默认为 true)

Built-In Endpoints (内置端点)

当 management 依赖添加到了你的项目中, 那么会默认开启如下内置 endpoints:

Table - Default Endpoints:

Endpoint URI Description
BeansEndpoint /beans Returns information about the loaded bean definitions in the application (see BeansEndpoint)
HealthEndpoint /health Returns information about the “health” of the application (see HealthEndpoint)
InfoEndpoint /info Returns static information from the state of the application (see InfoEndpoint)
LoggersEndpoint /loggers Returns information about available loggers and permits changing the configured log level (see LoggersEndpoint)
MetricsEndpoint /metrics Return the application metrics. Requires the micrometer-core configuration on the classpath
RefreshEndpoint /refresh Refreshes the application state (see RefreshEndpoint)
RoutesEndpoint /routes Returns information about URIs available to be called for your application (see RoutesEndpoint)
ThreadDumpEndpoint /threaddump Returns information about the current threads in the application

除此之外, management 依赖提供了如下默认不开启的内置 endpoints:

Table - Disabled Endpoints:

Endpoint URI Description
EnvironmentEndpoint /env Return information about the environment and its property sources (see EnvironmentEndpoint)
CachesEndpoint /caches Returns information about the caches and permits invalidating them (see CachesEndpoint)
ServerStopEndpoint /stop Shuts down the application server (see ServerStopEndpoint)

可通过设置 endpoints.all.sensitive: false 来开启所有端点的不授权访问, 但是这必须非常小心地使用, 因为私有的和敏感的信息会被暴露

Customization (自定义)

The beans endpoint is composed of a bean definition data collector and a bean data implementation.

beans endpoint 由一个 bean 定义数据收集器和一个 bean 数据实现组成

The bean definition data collector (BeanDefinitionDataCollector) is responsible for returning a publisher that returns the data used in the response.

bean 定义数据收集器 (BeanDefinitionDataCollector) 负责返回一个 publisher, 这个 publisher 返回响应中使用的数据

The bean definition data (BeanDefinitionData) is responsible for returning data about an individual bean definition.

bean 定义数据 (BeanDefinitionData) 负责返回关于单个 bean 定义的数据

To override the default behavior for either of the helper classes, either extend the default implementations (DefaultBeanDefinitionDataCollector, DefaultBeanDefinitionData), or implement the relevant interface directly.

要覆盖任一助手类的默认行为, 请扩展默认实现 (DefaultBeanDefinitionDataCollector, DefaultBeanDefinitionData), 或直接实现相关接口

To ensure your implementation is used instead of the default, add the @Replaces annotation to your class with the value being the default implementation.

为确保使用你的实现而不是默认实现, 将 @Replaces 注解添加到你的类上, 并将值作为默认实现

Customization (自定义)

health endpoint 包含一个 health aggregator 和多个 health indicators

要增加一个 health indicator, 创建一个实现 HealthIndicator 的 bean 类

建议还要使用 @Liveness 或 @Readiness 标识符

如果没有使用标识符, 那么这个 health indicator 将会是 /health 和 /health/readiness endpoints 的一部分

基类 AbstractHealthIndicator 可用于子类以使这个实现过程更加容易

所有的 health indicator beans 都被集合到 health aggregator

要提供你自己实现的 health aggregator, 创建一个实现 HealthAggregator 接口的类并将其注册为一个 bean

要确保你的实现替代了 health aggregator, 添加 @Replaces 注解到你的类上, 注解的值是默认的实现 DefaultHealthAggregator

默认的 health aggregator 返回一个根据指标的健康状况计算的整体状况

一个 health status 包含多种信息:

Name Meaning
Name The name of the status
Description The description of the status
Operational Whether the functionality the indicator represents is functional
Severity How severe the status. A higher number is more severe

The “worst” status is returned as the overall status

“worst` 状态作为整体状态返回

A non-operational status is selected over an operational status

在操作状态上选择 “非操作” 状态

A higher severity is selected over a lower severity

选择较高的严重性的而不是较低的严重性

The Metrics Endpoint (指标端点)

通过集成 Micrometer ,Micronaut 可以暴露应用程序指标

如果你使用 Micronaut CLI 创建项目, 添加 micrometer 的其中一个特性来开启指标并预配置选择的 registry 到你项目中:

mn create-app my-app --features micrometer-atlas

metrics endpoint 返回关于应用程序的指标的信息

要执行 metrics endpoint, 发送一个 GET 请求到 /metrics

这个 endpoint 返回一个可用指标名称的列表

你可以通过使用 /metrics/[name] 来获取特定的指标, 例如 /metrics/jvm.memory.used

查看文档 Micronaut Micrometer 来获取一个 registries 列表和关于如何配置, 暴露, 自定义 metrics 输出的信息

The Refresh Endpoint (刷新端点)

Refresh endpoint 刷新应用程序的状态, 导致所有上下文中的 Refreshable beans 被销毁并重新实例化直到下次请求

这通过在应用上下文中发送一个 RefreshEvent 来完成

要执行 refresh endpoint, 发送一个 POST 请求到 /refresh

curl -X POST http://localhost:8080/refresh

在没有发送 body 的情况下执行时, endpoint 首先刷新 Environment 并执行 diff 以检测任何更改, 然后仅在检测到更改时执行刷新

如果要不管环境变化, 跳过这个检查并刷新所有 @Refreshable beans (例如, 强制刷新来自第三方服务的缓存响应), 那么在 POST 请求 body 中添加一个 force 参数

curl -X POST http://localhost:8080/refresh -H 'Content-Type: application/json' -d '{"force": true}'

Customization (自定义)
