Application Configuration

声明:

  • 本文由本人翻译自官方文档
  • 部分内容可能会有修改或删减, 一切以官方文档为准.
  • 本文翻译时 Micronaut 版本为 3.4.2

默认情况下, 可以在 Java properties, YAML, JSON 或者 Groovy files 中提供配置

约定是搜索名称为 application.yml, application.properties, application.json 或者 application.groovy 的文件

还可以通过系统属性或环境变量来覆盖任何配置属性

每个配置源都使用 PropertySource 接口来建模, 并且这个机制是可以扩展的, 允许实现额外的 PropertySourceLoader 实现

The Environment (环境)

应用程序环境被建模为 Environment 接口, 这个接口允许在创建一个 ApplicationContext 时定义一个或多个唯一环境名称

Example - Initialing the Environment:

激活的环境名允许根据环境加载不同的配置文件, 还可以使用 @Requires 注解来条件化加载 Beans 或 Bean @Configuration 包

除此之外, Micronaut 还会尝试检测当前的环境, 例如在 Spock 和 JUnit 测试中, io.micronaut.context.env.Environment.TEST 是自动激活的环境

io.micronaut.context.env.Environment 接口定义了一些预定义好的环境名称

可以通过 micronaut.environments 系统属性或者 MICRONAUT_ENVIRONMENTS 环境变量来指定激活的环境, 多个环境名称使用逗号进行分隔:

Environment Priority (环境优先级)

Micronaut 基于指定的环境来加载属性源, 不同环境的属性源可能多个相同的属性名称, 使用环境的顺序来指明使用哪个属性值

Micronaut 使用如下优先级来处理(越下面优先级越高):

  • 推断的环境
  • 系统属性 micronaut.environments 设置的环境
  • 环境变量 MICRONAUT_ENVIRONMENTS 设置的环境
  • 通过应用程序上下文 Builder 明确指定的环境

Disabling Environment Detection (关闭环境检测)

可以通过将 系统属性micronaut.env.deduction 和 环境变量MICRONAUT_ENV_DEDUCTION 设置为 false 来关闭环境自动检测

关闭了环境自动检测, 但是还是会使用手动指定的环境

Externalized Configuration with PropertySources (使用 PropertySource 的外部化配置)

额外的 PropertySource 实例可以在初始化 ApplicationContext 之前被添加到环境中

Alternatively one can register a PropertySourceLoader by creating a META-INF/services/io.micronaut.context.env.PropertySourceLoader file containing a reference to the class name of the PropertySourceLoader.

Included PropertySource Loaders (已经包含的 PropertySource 加载器)

Micronaut 默认已经包含了一些从给定的位置和优先级加载属性的 PropertySourceLoader 实现

  1. 命令行参数
  2. 来自 SPRING_APPLICATION_JSON 的属性
  3. 来自 MICRONAUT_APPLICATION_JSON 的属性
  4. Java 系统属性
  5. 操作系统环境变量
  6. 按顺序从系统属性 micronaut.config.files 或者 环境变量 MICRONAUT_CONFIG_FILES 指定的配置文件中加载. 文件路径可以使用逗号来分隔多个, 按文件声明的先后顺序明确优先级. 文件可以从文件系统中获取, 也可以使用 classpath: 前缀从类路径中获取
  7. 从 application-{environment}.{extension} 中获取环境指定属性
  8. 从 application.{extension} 获取应用程序指定属性

默认支持 .properties,.json,yml, 还支持 .groovy

Using the @Property Annotation (使用 @Property 注解)

This is because @Value only resolves placeholders within the value specified to it.

@Value 注解必须使用 ${...} 语法来引用配置值, 如果传入普通字符串, 就只会将这个字符串注入到 Bean 属性中

Bootstrap Configuration (启动配置)

在一个 Application context 创建之前, 一个 bootstrap context 将会被创建, 用来存储必要的配置, 用来获取关于主 Context 的额外的配置

通常这些额外配置在一些远程源上

bootstrap context 是否开启取决于以下条件, 以下条件按顺序进行检查:

  • BOOTSTRAP_CONTEXT_PROPERTY 系统属性是否配置了
  • application context builder 选项 bootstrapEnvironment 是否有设置
  • BootstrapPropertySourceLoader Bean 是否存在, 通常这个 Bean 来自 micronaut-discovery-client 依赖

配置属性必须在 application context 配置文件解析之前提供, 例如在使用分布式配置时, 配置属性被保存在一个 bootstrap configuration file 中

一旦明确了 bootstrap context 是开启的, bootstrap 配置文件将会使用普通 application 配置的规则进行读取, 唯一的不同是前缀(bootstrap 代替 application)

文件名称前缀 bootstrap 使用系统属性 micronaut.bootstrap.name 来配置

bootstrap context configuration 自动被带到 main context, 所以没有必要在 main context 中重复这些配置属性

bootstrap context configuration 比 main context configuration 有更高的优先级, 这意味着如果一个配置属性要在两个 context 出现, 将这个属性配置放到 bootstrap context configuration 中

Bootstrap Context Beans (Bootstrap 上下文中的 Bean)

一个 Bean 要被解析在 bootstrap context 中, 这个 Bean 必须使用 @BootstrapContextCompatible 注解来修饰

特别是任何参与到获取分布式配置的 Bean 必须要使用这个注解
