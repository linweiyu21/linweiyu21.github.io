Bean Qualifiers Bean (限定符)

当你需要注入一个有多个实现的接口的 Bean 时, 你需要使用一个 qualifier(限定词, 限定语)

Qualifying By Name (通过名字来限定)

@Named qualifier 的值 v8 加上 类Engine 构成 v8Engine, 再使用大小写不敏感的判断: v8Engine == V8Engine, 所以注入的是 Engine 的 V8Engine 实现

还可以在类上使用 @Named 来明确声明这个 Bean 的名称, 以便在注入时使用这个名称

通过使用 @Qualifier 注解来声明自定义的注解来进行 qualify

Limiting Injectable Types (限制注入的类型)

使用 @Bean 注解来限定允许注入的接口或类, 而不是具体的类型

@Bean(typed = Engine) 限定了只能使用 Engine 接口来进行依赖注入, 而不能使用 V8Engine 类来进行依赖注入

V8Engine 类必须实现 @Bean 注解中声明的类或接口, 否则将会产生编译时异常

Scopes (作用域)

Built-In Scopes (内建的作用域)

Table - Micronaut Built-in Scopes

Type Description
@Singleton Singleton scope indicates only one instance of the bean will exist
@Context Context scope indicates that the bean will be created at the same time as the ApplicationContext (eager initialization)
@Prototype Prototype scope indicates that a new instance of the bean is created each time it is injected
@Infrastructure Infrastructure scope represents a bean that cannot be override or replaced using @Replaces because it is critical to the functioning of the system
@ThreadLocal @ThreadLocal scope is a custom scope that associates a bean per thread via a ThreadLocal
@Refreshable @Refreshable scope is a custom scope that allows a bean’s state to be refreshed via the /refresh endpoint
@RequestScope @RequestScope scope is a custom scope that indicates a new instance of the bean is created and associated with each HTTP request

@Prototype 注解是 @Bean 注解的同义词, 因为默认的 scope 是 prototype 的

可以通过定义一个实现了 CustomScope 接口的 @Singleton Bean 来添加额外的 scope

默认情况下, 在启动一个 ApplicationContext 时, 将会延迟与按需创建 @Singleton scope 的 Bean

可以使用 @Context 注解来将 Bean 的生命周期绑定到 ApplicationContext 上, 换句话说, 就是在 ApplicationContext 启动的时候创建这个 Bean

或者, 使用 @Parallel 注解来修饰一个 @Singleton scope 的 Bean, 允许并行初始化这个 Bean

如果你的 Bean 在并行初始化的时候失败了, 那么整个 Application 将会自动关闭

Bean Events (Bean 事件)

你可以使用如下接口来挂接(hook)到 Beans 的创建中:

  • BeanInitializedEventListener : 允许属性都设置后之后, 在 @PostConstruct 事件 hooks 之前修改和替代一个 Bean

BeanInitializedEventListener 接口通常和 Factory Bean 结合使用

Bean 事件监听器在 类型转换器(type converters) 之前进行初始化. 如果你的监听器依赖类型转换, 或者依赖配置属性Bean, 或其他任何机制, 你可能会得到一个关于类型转换的异常

Write an AnnotationMapper to Introspect Existing Annotations (编写一个 AnnotationMapper 来内省一个已存在的注解)

如果你想默认为一个已存在的注解进行内省, 你可以编写一个 AnnotationMapper

一个例子是 EntityIntrospectedAnnotationMapper, 这个 Mapper 确保了所有使用 @javax.persistence.Entity 注解的 Beans 都默认被内省了

AnnotationMapper 必须存在于 annotation processor classpath

The BeanWrapper API (BeanWrapper API)

一个 BeanProperty 提供了读写类中一个属性值的原始访问方式, 没有提供任何自动类型转换

传递给 set/get 方法的值应该与底层属性类型相匹配, 否则会发生异常

为了提供额外的类型转换智能, BeanWrapper 接口允许包装现有的 Bean 实例, 读写 Bean 属性, 并在必要时执行类型转换

Jackson and Bean Introspection (Jackson 和 Bean 内省)

Jackson 被配置为使用 BeanIntrospection API 来读写属性值与构造对象, 结果就是无反射的序列化/反序列化

从性能的角度来看, 这是有益的, 并且需要更少的配置来正确地与运行时操作, 例如 GraalVM native

Bean Validation (Bean 校验)

从 Micronaut 1.2 开始, Micronaut 内置了校验那些使用了 javax.validation 注解修饰的 Beans 的支持

Micronaut 现在的实现不是完全遵循 Bean Validator 规范的, 因为这个规范重度依赖基于反射的 APIs

现在还不支持如下特性:

  • 泛型参数类型上的注解, 因为现在只有 Java 支持这个特性
  • 任何与 constraint metadata API 的交互, 因为 Micronaut 使用编译时生成的元数据
  • 基于 XML 的配置
  • 使用 io.micronaut.validation.validator.constraints.ConstraintValidator 来代替 javax.validation.ConstraintValidator, Micronaut 的 ConstraintValidator 支持编译时的校验注解
