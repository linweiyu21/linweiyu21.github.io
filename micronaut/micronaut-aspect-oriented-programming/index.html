Aspect Oriented Programming

Generally AOP can be thought of as a way to define cross-cutting concerns (logging, transactions, tracing, etc.) separate from application code in the form of aspects that define advice.

通常来说 AOP 可以被认为是定义横切关注点(日志, 事务, 跟踪等)的一种方式, 它以定义通知的方面的形式与应用程序代码分开

通常有两种形式的 通知(advice):

  • 环绕通知(Around Advice), 包装一个方法或类
  • 引导通知(Introduction Advice), 引入一个新的行为到一个类中

在现代的 Java 应用程序中, 声明 通知(advice) 通常使用注解的形式. Java 世界中最出名的注解通知(advice)可能是 @Transactional, 这个注解标定了 Spring 和 Grails 应用程序中的事务边界

传统 AOP 实现的缺点是非常依赖运行时代理创建和使用反射, 拖慢了应用程序的性能, 使调试更加困难, 增加了内存消耗

Micronaut 尝试解决这些问题, 通过提供一个简单的不使用反射的编译时 AOP API

Customizing Proxy Generation (自定义代理的生成)

The default behaviour of the Around annotation is to generate a proxy at compile time that is a subclass of the proxied class.

Around 注解的默认行为是在编译时生成一个代理, 这个代理是被代理类的一个子类

换句话说, 在上面的例子中, 一个编译时的 NotNullExample 类的子类会被创建, 它的代理方法使用拦截器处理逻辑进行包装, 方法的原始行为通过调用 super 来执行

这种行为更加高效, 因为只需要一个 Bean 实例, 但是在不同的应用场景下, 你可能希望修改这种行为

@Around 注解支持多个属性来允许你修改这个行为, 包括:

  • proxyTarget, 默认为 false, 如果设置为 true, 代理将会委托给原始的 Bean 实例, 而不是委托给一个子类来调用 super
  • hotswap, 默认为 false, 和 proxyTarget=true 一样, 但除此之外代理还会实现 HotSwappableInterceptedProxy 来包装每个方法在一个 ReentrantReadWriteLock 中, 并允许在运行时切换(swapping)目标实例(target instance)
  • lazy, 默认为 false, 默认情况下 Micronaut 在 proxy 被创建的时候 eagerly 初始化 proxy target, 如果设置为 true 那么 proxy target 将会被延迟地解析, 在每个方法调用的时候

AOP Advice on @Factory Beans (在 @Factory Beas 上的 AOP 通知)

Bean Factories 上的 AOP advice 语义和不同的类是不一样的:

  1. 使用于 @Factory Bean 类级别上的 AOP advice 将 advice 作用于 Factory 本身, 而不是 Factory 中使用 @Bean 注解修饰的任何 Bean
  2. 使用于一个使用了作用域修饰的方法上的 AOP advice 将会应用由 Factory 提供的 AOP advice

The rationale for this behaviour is that you may at times wish to apply advice to a factory and at other times apply advice to the bean produced by the factory.

这种行为的基本原理是, 有时你可能希望将 advice 应用到一个 Factory, 而有时则希望将 advice 应用到 Factory 产生的 Bean 上

现在不能将方法级别的 advice 应用到 @Factory Bean 上, 所有用于 Factory 的 advice 必须应用在类级别上

你可以通过将方法定义为 non-public 来控制哪个方法是否要应用 advice, non-public 说明方法不应用 advice

Bean Life Cycle Advice (Bean 生命周期 Advice)

有 3 种方式可以将 Advice 应用到 Bean 的生命周期中:

  • 拦截 Bean 的创建
  • 拦截 Bean 的 @PostConstruct 调用
  • 拦截 Bean 的 @PreDestroy 调用

Micronaut 通过定义额外的 @InterceptorBinding 元注解来支持这 3 种场景

1 : The @AroundConstruct annotation is added to indicate that interception of the constructor should occur

2 : An @InterceptorBinding definition is used to indicate that @PostConstruct interception should occur

3 : An @InterceptorBinding definition is used to indicate that @PreDestroy interception should occur

4 : The bean is defined as @Prototype so a new instance is required for each injection point

Cache Advice (缓存 Advice)

Micronaut 在 io.micronaut.cache 包中提供缓存注解

CacheManager 接口允许根据需要插入不同的缓存实现

SyncCache 接口提供一个同步的 API 用来缓存, 同时 AsyncCache API 允许非阻塞的操作

Cache Annotations (缓存注解)

支持如下缓存注解:

  • @Cacheable : 指明一个方法在指定缓存中是可缓存的
  • @CachePut : 指明一个方法调用的返回值应该被缓存起来. 不像 @Cacheable 那样, 方法的执行永远不会被跳过
  • @CacheInvalidate: 指明方法的调用应该导致一个或多个缓存无效

使用上面其中一个注解会激活 CacheInterceptor

如果方法的返回类似是非阻塞类型, 则缓存生成的结果

如果缓存实现支持非阻塞操作, 那么缓存值可以不阻塞的方式进行读取

Naming Caches (命名缓存)

在 micronaut.caches 下使用 kebab-case 方式定义缓存的名称

如果你使用了驼峰命名, 这个驼峰名称将会被转换为 kebeb-case

例如, myCache 转换为 my-cache

kebab-case 形式的名称必须在 @Cacheable 注解中用来引用缓存

要配置一个和 maximumWeight 配置一起使用的 weigher, 创建一个实现 io.micronaut.caffeine.cache.Weigher 的 Bean

要关联一个给定的 weigher 到一个指定的缓存上, 使用 @Named(<cache name>) 注解来修饰这个 weigher Bean

没有 named qualifier 的 weigher 将会应用到所有没有一个 named weigher 的缓存中

如果没有 weigher Bean, 那么将会使用一个默认实现

Dynamic Cache Creation (动态缓存创建)

在缓存不能提前配置的情况下, 可以注册一个 DynamicCacheManager Bean

当尝试获取一个还没有提前定义的缓存时, 动态缓存管理器将会被调用来创建并返回一个缓存

默认情况下, 如果没有其他动态缓存管理器定义在应用中, Micronaut 注册一个 DefaultDynamicCacheManager 实例使用默认值来创建 Caffeine 缓存

Retry Advice (重试 Advice)

Micronaut 提供了 Retryable 注解来进行操作重试

@Retryable 注解默认会重试三次, 每次重试之间的使用一秒的指数级进行延迟

第一次尝试使用 1 秒延迟, 第二次延迟使用 2 秒延迟, 第三次延迟使用 3 秒延迟

Circuit Breaker (断路器)

Circuit Breaker 模式的设计目的是通过允许一定数量的失败请求, 然后就打开一个断路, 该断路在允许额外的的重试尝试之前保持打开一段时间, 从而解决失败重试请求过多的问题

@CircuitBreaker 注解是 @Retryable 注解的变体, 这个注解有一个 reset 成员属性来指定断路器在重置之前要打开多久, 默认是 20 秒

Factory Bean Retry (工厂 Bean 重试)

当 @Retryable 应用到 Bean 工厂方法上时, 它的行为就像将这个注解放在要返回的类型上

重试行为将会在返回对象上的方法被调用时被应用

Bean Factory 本身是不可以重试的, 如果你想要实现重试创建 Bean 的功能, 你应该将这个行为委托给另一个使用了 @Retryable 注解的单例Bean
