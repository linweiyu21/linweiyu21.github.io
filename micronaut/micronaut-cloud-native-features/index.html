Cloud Native Features (云原生特性)

声明:

  • 本文由本人翻译自官方文档
  • 部分内容可能会有修改或删减, 一切以官方文档为准.
  • 本文翻译时 Micronaut 版本为 3.4.2

The majority of JVM frameworks in use today were designed before the rise of cloud deployments and microservice architectures.

今天使用的大多数 JVM 框架都是在云部署和微服务架构兴起之前设计的

使用这些框架构建的应用都是为了部署在传统 Java 容器中的

所以, 这些框架中的云支持通常作为附加组件而不是核心设计功能提供

Micronaut 为了构建用于云的微服务而从零开始设计. 因此, 你的应用本身可以使用许多通常需要外部库或服务的关键功能

To override one of the industry’s current favorite buzzwords, Micronaut applications are “natively cloud-native”.

为了超越业界当前最流行的框架之一, Micronaut 应用程序是 “本地化云原生” 的

下面是一些已经直接集成在 Micronaut 运行时的特定于云的功能 (cloud-specific features):

  • 分布式配置 (Distributed Configuration)
  • 服务发现 (Service Discovery)
  • 客户端侧的负载均衡 (Client-Side Load-Balancing)
  • 分布式追踪 (Distributed Tracing)
  • 无服务功能 (Serverless Functions)

Cloud Configuration (云配置)

为云构建的应用程序通常需要适应在云环境中运行, 以分布式方式读取和共享配置, 并在必要时将配置外部化到环境 (environment) 中

Micronaut’s Environment concept is by default Cloud platform-aware and makes a best effort to detect the underlying active environment.

Micronaut 的 Environment 概念默认是云平台感知的, 并尽最大努力检测底层激活的环境 (environment)

然后可以使用 @Requires 注解来 条件化加载 bean 定义. 即在对应的云环境中使用 @Requires 注解来激活对应的环境支持

下表中总结了 Environment 接口中的常量并提供一个例子:

Table - Micronaut Environment Detection:

Constant Description Requires Example Environment name
ANDROID The application is running as an Android application @Requires(env = Environment.ANDROID) android
TEST The application is running within a JUnit or Spock test @Requires(env = Environment.TEST) test
CLOUD The application is running in a Cloud environment (present for all other cloud platform types) @Requires(env = Environment.CLOUD) cloud
AMAZON_EC2 Running on Amazon EC2 @Requires(env = Environment.AMAZON_EC2) ec2
GOOGLE_COMPUTE Running on Google Compute @Requires(env = Environment.GOOGLE_COMPUTE) gcp
KUBERNETES Running on Kubernetes @Requires(env = Environment.KUBERNETES) k8s
HEROKU Running on Heroku @Requires(env = Environment.HEROKU) heroku
CLOUD_FOUNDRY Running on Cloud Foundry @Requires(env = Environment.CLOUD_FOUNDRY) pcf
AZURE Running on Microsoft Azure @Requires(env = Environment.AZURE) azure
IBM Running on IBM Cloud @Requires(env = Environment.IBM) IBM
DIGITAL_OCEAN Running on Digital Ocean @Requires(env = Environment.DIGITAL_OCEAN) digitalocean
ORACLE_CLOUD Running on Oracle Cloud @Requires(env = Environment.ORACLE_CLOUD) oraclecloud

注意, 你可以激活多个环境, 例如, 运行在 AWS 上的 Kubernetes 中

除此之外, 使用上表中常量名称, 你可以创建用于指定环境的配置文件, 例如, 创建了一个 src/main/resources/application-gcp.yml 文件, 这个文件就只会在运行于 Google Compute 时加载

Any configuration property in the Environment can also be set via an environment variable. For example, setting the CONSUL_CLIENT_HOST environment variable overrides the host property in ConsulConfiguration.

Environment 中的任何配置属性可以通过一个环境变量来设置. 例如, 设置 CONSUL_CLIENT_HOST 环境变量来覆盖 ConsulConfiguration 中的 host 属性

Using Cloud Instance Metadata (使用云实例元数据)

当 Micronaut 检测到它在支持的云平台上运行时, 它会在启动时填充接口 ComputeInstanceMetadata

As of Micronaut 2.1.x this logic depends on the presence of the appropriate core Cloud module for Oracle Cloud, AWS, or GCP.

从 Micronaut 2.1.x 开始, 这个逻辑取决于是否存在 用于Orcale Cloud,AWS或GCP的 适当的云模块核心

所有这些数据都合并到正在运行的 ServiceInstance 的 metadata 属性中

If you connect remotely via a client, the instance metadata can be referenced once you have retrieved a ServiceInstance from either the LoadBalancer or DiscoveryClient APIs.

如果你通过 client 进行远程连接, 则可以在你从 LoadBalancer 或 DiscoveryClient API 检索到 ServiceInstance 后引用实例元数据

The Netflix Ribbon client-side load balancer can be configured to use the metadata to do zone-aware client-side load balancing.

Distributed Configuration (分布式配置)

As you can see, Micronaut features a robust system for externalizing and adapting configuration to the environment inspired by similar approaches in Grails and Spring Boot.

正如你所见, Micronaut 有一个强大的系统来外部化和可调整的配置

如果要在多个微服务中共享配置, Micronaut 包含了分布式配置的的 APIs

ConfigurationClient 接口有一个 getPropertySources 方法, 可以实现这个方法来读取解析来自分布式源的配置

getPropertySources 方法返回一个输出0个或多个 [PropertySource](https://docs.micronaut.io/latest/api/io/micronaut/context/env/PropertySource.html 的 Publisher

默认的实现是 DefaultCompositeConfigurationClient, 这个 client 合并所有注册的 ConfigurationClient beans 到单独一个 bean 中

你可以实现你自己的 ConfigurationClient, 或者使用 Micronaut 提供的实现

Client Side Load Balancing (客户端侧负载均衡)

当从 Consul, Eureka, 或其他服务发现 servers 中 discovering services 时, DiscoveryClient 输出一个可用的 ServiceInstance 列表

Micronaut by default automatically performs Round Robin client-side load balancing using the servers in this list.

Micronaut 默认使用这个列表中的 servers 执行客户端侧轮询负载均衡

This combined with Retry Advice adds extra resiliency to your Microservice infrastructure.

这个和 Retry Advice 相结合, 为你的微服务基础架构增加了额外的弹性

由 LoadBalancer 接口处理负载均衡, 这个接口有一个 LoadBalancer.select() 方法来返回一个 Publisher, 这个 Publisher 输出一个 ServiceInstance

返回 Publisher 是因为选择一个 ServiceInstance 的过程可能导致一个网络操作, 取决于所采用的 Service Discovery 策略

LoadBalancer 接口的默认实现是 DiscoveryClientRoundRobinLoadBalancer. 你可以使用另一个实现来代替这个策略, 这个实现自定义了客户端侧负载均衡在 Micronaut 中是怎样处理的, 因为有许多不同方式来优化负载均衡

例如, 你可能希望在特定区域中的服务之间进行负载均衡, 或者在具有最佳整体响应时间的服务器之间进行负载均衡

要代替 LoadBalancer, 定义一个 Bean 来 replaces DiscoveryClientLoadBalanceFactory. 实际上, 这正是 Netflix Ribbon 支持所做的事情
